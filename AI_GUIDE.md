# BOOONG Design System AI Guide

Use this guide when an AI assistant works on a BOOONG-based project.

## Rules

- Use BOOONG design tokens before introducing new raw colors, spacing, radii, shadows, or font stacks.
- Keep these tokens stable unless a design-system release intentionally changes them: `--brand`, `--brand-deep`, `--brand-soft`, `--brand-ink`, `--bg`, `--text`, `--border`, `--radius`, `--space-*`, `--font-*`.
- Reuse existing component classes before creating project-specific variants.
- Project CSS should define product-specific layout, page composition, and domain visuals.
- Do not redefine brand colors, typography scale, spacing scale, or radii inside consumer projects.
- Preserve legacy classes generated by BOOONG dashboard JavaScript, including `.block`, `.callout`, `.question`, `.case`, `.answer`, and `.material`.

## Mobile Overlay Components

These components exist for UIs where a map, canvas, image, or full-viewport content sits behind interactive controls. Use them when the content layer must not be interrupted by a sidebar or a stacked panel.

### When to apply the overlay pattern

Apply when **all three** conditions hold:
1. The primary content is spatial or visual (map, canvas, media viewer, gallery).
2. The user must interact with the content and the controls simultaneously — scrolling to reach controls is not acceptable.
3. The viewport is narrow (≤ 620 px) **or** the product is intentionally a full-screen tool at any size.

Do not apply to document-style pages, dashboards with data tables, or any layout where the user reads top-to-bottom. A sidebar or stacked panel is correct there.

---

### `.overlay-bar` — Floating control bar

A semi-transparent, backdrop-blurred bar that floats above content. Use it for persistent controls that must stay reachable while the user interacts with the layer beneath.

```html
<!-- Top: title + action -->
<header class="overlay-bar overlay-bar--top overlay-bar--sticky-top">
  <span>지도 제목</span>
  <button class="btn--round btn--ghost">?</button>
</header>

<!-- Bottom: slider or primary action -->
<footer class="overlay-bar overlay-bar--bottom overlay-bar--sticky-bottom">
  <span>06시</span>
  <input type="range" />
</footer>
```

**Principles**
- Always pair `overlay-bar--sticky-top` / `overlay-bar--sticky-bottom` with a `position: relative` parent that fills the viewport (`height: 100dvh`).
- `overlay-bar--bottom` includes `env(safe-area-inset-bottom)` padding automatically. Do not add extra bottom padding on top of it.
- Keep content inside a bar to **one row**. If you need two rows, reconsider the information hierarchy — split into two bars or move secondary controls elsewhere.
- Use `--z-sticky` (200) for bars. If a modal or dropdown must appear above the bar, those already use `--z-modal` (400) and `--z-dropdown` (100) respectively.
- Do not use `.overlay-bar` for a fixed site header on a scrolling page. Use a regular `<header>` with `position: sticky` there.

---

### `.chip-toggle` — Toggle chip

A pill-shaped toggle button. Use it for **binary layer or filter switches** rendered as floating chips above a visual content layer — not for navigation, not for form inputs.

```html
<!-- Standalone -->
<button class="chip-toggle" type="button" aria-pressed="false" data-layer="density">
  인구밀도
</button>

<!-- Group (horizontal) -->
<div class="chip-toggle-group" role="group" aria-label="레이어 선택">
  <button class="chip-toggle is-active" type="button" aria-pressed="true" data-layer="flow">유동인구</button>
  <button class="chip-toggle" type="button" aria-pressed="false" data-layer="density">밀도</button>
  <button class="chip-toggle is-active" type="button" aria-pressed="true" data-layer="subway">지하철</button>
</div>

<!-- Group (vertical, right-aligned overlay) -->
<div class="chip-toggle-group chip-toggle-group--vertical">
  ...
</div>
```

**Principles**
- Always set `aria-pressed` and update it in JavaScript when the state changes.
- Use `data-layer` or equivalent attributes to drive JS logic — do not rely on class inspection.
- `.chip-toggle` is **not** a replacement for `.btn--primary` or form `<input type="checkbox">`. Use it only when the control floats above a visual layer on mobile or as a compact inline filter.
- When the same toggle also appears in a desktop sidebar (e.g., as a `.layer-toggle`), sync `aria-pressed` and `.is-active` across both elements via `querySelectorAll('[data-layer="…"]')`. Never duplicate state management.
- Limit groups to **4 chips** max per row. Beyond that, switch to a dropdown or a panel.

**`.chip-toggle` vs. `.chip` (existing)**

| | `.chip` | `.chip-toggle` |
|---|---|---|
| Purpose | Tag, label, removable filter badge | Binary on/off toggle |
| Interaction | Has `chip__remove` for deletion | Clicked to flip `aria-pressed` |
| Default background | Brand blue (always filled) | White semi-transparent / brand when active |
| Use on overlays | No | Yes |

### `.btn--round` — Compact round icon button

A 32 px or 44 px circular button. Use it for single-glyph actions (close, help, back) inside `.overlay-bar` or in tight spaces where the standard `.btn` height (44 px) is too tall.

```html
<!-- 32px: inside a compact overlay bar -->
<button class="btn btn--ghost btn--round" type="button" aria-label="도움말">?</button>

<!-- 44px: standalone floating action -->
<button class="btn btn--primary btn--round-lg" type="button" aria-label="현재 위치">◎</button>
```

**Principles**
- Always provide `aria-label`. A round button contains only a glyph; screen readers need the label.
- Use `.btn--round` (32 px) only inside a bar or a compact header where vertical space is the constraint. Prefer `.btn--round-lg` (44 px) for standalone floating buttons — it meets the minimum touch-target size.
- Apply a variant class (`.btn--ghost`, `.btn--primary`, `.btn--secondary`) alongside `.btn--round`. The round modifier only sets dimensions.
- Do not use `.btn--round` for primary page-level actions. A labeled `.btn--primary` is clearer.

---

### `.info-float` — Contextual floating info card

A small card that surfaces the name and a key metric for a selected item, rendered as an overlay on a visual content layer.

```html
<div class="info-float" aria-live="polite">
  <strong class="info-float__title">수성구 범어2동</strong>
  <span class="info-float__sub">12,430 명/km²</span>
</div>
```

**Principles**
- Position with `position: absolute` inside the content's `position: relative` container. Use CSS variables for offset so the card doesn't collide with `.overlay-bar--top` or other pinned elements.
- Add `aria-live="polite"` so screen readers announce updates when the user selects a new item.
- Show at most **two lines**: a title and one supporting metric. If more data is needed, open a sheet or panel instead.
- Hide with `.is-hidden` (`display: none`) when no item is selected. Do not leave an empty card visible.
- Do not use `.info-float` for persistent data (e.g., a legend or a tooltip that appears on hover). It is for **selected-state** information only.

### Safe area and full-viewport layout

When building a full-viewport tool (map, viewer, game), set the root container to `100dvh` and use `env(safe-area-inset-*)` for padding on fixed edges.

```css
/* Consumer project CSS — not in BOOONG itself */
.app-shell--fullscreen {
  height: 100dvh;
  overflow: hidden;
}

/* If overlay-bar--bottom is not used and you need manual safe area: */
.custom-bottom-bar {
  padding-bottom: max(var(--space-3), calc(var(--space-3) + var(--safe-bottom)));
}
```

**Principles**
- Use `100dvh` (dynamic viewport height), not `100vh`, on mobile. `100vh` does not account for the browser chrome on iOS Safari.
- `--safe-bottom` and `--safe-top` are defined in the token layer. Reference them instead of writing `env(safe-area-inset-bottom, 0px)` inline.
- Never apply full-viewport layout to a document or dashboard page. It is appropriate only when the primary content is spatial and must fill the screen edge-to-edge.
- When hiding a sidebar at mobile breakpoints, do not set `display: none` on the sidebar alone — also ensure the main content area expands to fill the freed space (use `grid-template-columns: 1fr` or `display: block`).

## Classroom Game UX — Device Principles

These principles apply to BOOONG-based interactive classroom games (map redistricting, trading simulations, card-selection exercises, and similar student activities). They are derived from three game types and cover the full device range students actually use.

### Device tiers

BOOONG uses three breakpoints that map cleanly to device classes:

| Tier | Width | Typical device | Layout mode |
|------|-------|---------------|-------------|
| **Mobile** | ≤ 760 px | Smartphone (portrait) | Single column, full-width |
| **Tablet** | 761 – 900 px | iPad / tablet (portrait) | Two-column cards possible |
| **PC** | ≥ 901 px | Desktop, tablet (landscape) | Multi-column, sidebar |

Tablet in **portrait** (768–834 px) falls in the tablet tier and still needs most mobile principles. Tablet in **landscape** (1024 px+) falls in the PC tier and gets the full desktop layout automatically.

---

### Universal principles (all tiers)

These apply regardless of device and should never be omitted.

- **`overlay-bar--top` for the persistent status bar.** Always show group name, current phase/round, and any countdown timer in a top overlay bar. It must not scroll away.
- **44 px minimum touch target.** Applies to all tiers — tablet and PC users also tap. Never shrink `.btn`, `.field__input`, or interactive cards below 44 px in height.
- **`.toast` for teacher-triggered phase transitions.** When the teacher advances the round or opens submission, notify students with a toast — never rely on a page reload.
- **`.alert-dialog` for dramatic reveal moments.** Role assignment, round results, and any high-stakes reveal should use the full modal treatment, not an inline message.
- **Submission CTA disabled until valid.** Block the submit button with `[disabled]` until all required inputs are complete (ratio sum = 100 %, minimum text length, all cards selected, etc.). Show an inline validation hint — do not wait for a server error.

---

### Status bar

```html
<!-- All game types, all tiers -->
<header class="overlay-bar overlay-bar--top overlay-bar--sticky-top">
  <span class="chip chip--gray">1모둠</span>
  <span>헌법 제정  2 / 5라운드</span>
  <span class="timer" data-state="normal">02:47</span>  <!-- add data-state="urgent" under 30 s -->
</header>
```

- Show at most three pieces of information: identity (group name), progress (phase / round), and urgency (timer if present).
- When a countdown reaches 30 seconds, switch `data-state` to `"urgent"` and apply `color: var(--color-error)`. Use a CSS attribute selector — do not add a second class.
- On **mobile**, keep label text to ≤ 6 characters per segment to avoid overflow.
- On **PC**, the status bar may be part of a regular sticky `<header>` instead of `.overlay-bar` if the page is document-style (not full-viewport).

---

### Submission CTA

```html
<!-- Mobile / Tablet portrait: sticky full-width -->
<footer class="overlay-bar overlay-bar--bottom overlay-bar--sticky-bottom">
  <button class="btn btn--primary" style="flex: 1" disabled>
    이번 라운드 투자 확정하기
  </button>
</footer>

<!-- PC: inline at end of scrollable content -->
<button class="btn btn--primary" style="width: 100%; margin-top: var(--space-6)" disabled>
  이번 라운드 투자 확정하기
</button>
```

- On **mobile and tablet portrait**, pin the button with `overlay-bar--bottom`. The bar includes `env(safe-area-inset-bottom)` automatically.
- On **PC**, placing the button inline at the bottom of the scroll is sufficient — the screen is tall enough that the button is rarely out of view.
- Always use full width (`flex: 1` or `width: 100%`) for a game CTA. This is not a paired-button dialog.

### Game type A — Spatial / map games (e.g. 게리맨더링)

The primary content is a map or canvas. Controls float above it.

| | Mobile | Tablet portrait | PC |
|--|--------|----------------|-----|
| Map area | `100dvh`, `touch-action: none` | `100dvh`, `touch-action: none` | `100dvh` or fixed height |
| Controls | `overlay-bar--bottom` | `overlay-bar--bottom` | Side panel or `overlay-bar` |
| Mode toggle | `.chip-toggle-group` (≤ 4) | `.chip-toggle-group` | `.chip-toggle-group` or sidebar buttons |
| District labels | `clamp(11px, 2vw, 13px)` | `clamp(12px, 1.5vw, 14px)` | `14px` |

On **tablet landscape / PC**, consider moving the mode-toggle group into a side panel if more than two modes exist. The `overlay-bar--bottom` is primarily a narrow-screen pattern.

Set `touch-action: none` on the map element to prevent the browser's default pan-and-zoom from interfering with drawing interactions.

---

### Game type B — Long-form input games (e.g. 환율 파도타기)

The primary interaction is reading content and entering values (sliders, textarea).

**Sliders**

Always pair every `<input type="range">` with a synchronized number input:

```html
<div class="field">
  <label class="field__label">원화 비중</label>
  <div style="display: flex; gap: var(--space-3); align-items: center">
    <input type="range" min="0" max="100" value="65"
           style="flex: 1; height: 6px" />          <!-- track height ≥ 6px -->
    <input type="number" class="field__input" min="0" max="100" value="65"
           style="width: 4.5rem; text-align: center" inputmode="numeric" />
  </div>
</div>
```

- On **mobile**, the number input is the reliable fallback when slider precision is too low.
- On **tablet and PC**, sliders are accurate enough to use directly; the number input can be secondary.
- The sum validation ("합계 100 %") must update on every `input` event, not on `change`.

**News / reading content**

| | Mobile | Tablet portrait | PC |
|--|--------|----------------|-----|
| News layout | 1 column, `.expandable` per article | 2 columns or `.expandable` | 2 columns side-by-side |
| Expandable default | Collapsed | Expanded | Expanded |

**Keyboard overlap**

When a `<textarea>` receives focus on mobile or tablet, the on-screen keyboard can cover the sticky submit button. Use the `visualViewport` API to shift the pinned footer upward:

```js
visualViewport.addEventListener('resize', () => {
  const offset = window.innerHeight - visualViewport.height;
  footer.style.transform = `translateY(-${offset}px)`;
});
```

This is outside BOOONG's scope and must be implemented in the consumer project.

---

### Game type C — Card selection games (e.g. 헌법 제정)

The primary interaction is choosing one option from a set of labeled cards per policy category.

**Card grid layout**

| | Mobile | Tablet portrait | PC |
|--|--------|----------------|-----|
| Cards per row | 1 | 2 | 2–3 |
| Show all sections | Stepper recommended | All visible | All visible |

The stepper pattern on mobile (one policy section at a time, "다음" to advance) reduces scroll depth and prevents students from accidentally skipping a section. On tablet and PC, show all sections at once — the screen is tall enough to orient the user.

Selected-state styling must go beyond a color change on the border. Fill the card background and add a visible checkmark or indicator:

```css
/* Consumer project CSS */
.policy-card[aria-selected="true"] {
  background: var(--color-primary-subtle);
  border-color: var(--color-primary);
  outline: 2px solid var(--color-primary);
}
```

Use `aria-selected` on `role="option"` cards inside a `role="listbox"` container for correct screen-reader semantics.

**Countdown timer urgency**

The constitution game has a live timer. Apply the urgency state to the status bar only — do not flash the entire page or interrupt the student's input.

| Need | BOOONG component | Tier |
|------|-----------------|------|
